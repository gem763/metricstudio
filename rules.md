# Codex Rules

## Language
- 모든 설명과 답변은 **한국어**로 작성합니다.
- 코드, 라이브러리/함수명은 원문(영어)을 유지합니다.
- 주석, docstring, 에러 메시지는 가능하면 한글로 작성합니다.

## Style
- 단계별로 설명합니다.
- 실행 가능한 코드/커맨드를 우선 제공합니다.
- 추상적인 설명보다 실무 중심으로 작성합니다.

## Coding
- 예시는 최소 단위로 제공합니다.
- 복사-붙여넣기 가능한 형태를 유지합니다.
- 코드는 항상 단순하게 유지하고, 불필요한 호환성 코드는 넣지 않습니다.
- 네이밍은 직관적으로 짧게 짓고, 의미가 바로 보이게 유지합니다.
- 중복/혼동되는 이름(예: 같은 의미의 속성 2개)은 하나로 통일합니다.
- API는 명시적이고 예측 가능하게 설계합니다(숨은 동작 최소화).
- 기본값/동작은 "없으면 없음" 원칙을 우선합니다(불필요한 자동 대체 지양).
- 지연 import보다 상단 import를 선호합니다(코드 단순성 우선).
- docstring 포맷은 """ 시작/종료를 각각 독립 줄에 두는 스타일을 선호합니다.
- 함수/구조가 너무 많아지면 통합해 인터페이스를 줄입니다.
- 책임에 맞는 계층에 로직을 배치합니다(데이터 정제/필터는 가능한 데이터 로드 계층).
- 함수를 직접 불러와서 사용하는 것보다는, 클래스를 만들고 구조화하여 사용하는 것을 선호합니다.

## Runtime / UX
- 진행률 표시(tqdm)는 실제로 시간이 걸리는 구간에 맞춰 보여줍니다.
- 시각화 결과는 사용자 관점의 가독성/일관성(범례 순서, 레이블, 축 표시)을 우선합니다.
- OS/환경 차이(Windows/WSL/macOS)까지 고려해 동작하게 만듭니다.
